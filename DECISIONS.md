# 개발 결정 기록

> **2026-06 포지셔닝 업데이트:** 프로젝트 일차 정체성이 "수학적 추론의 형식 검증 프레임워크"로 재정립됐다. 아래의 역사적 결정들(언어·산출물·철학적 배경)은 그대로 보존하되, 논리적 다원주의(유리학)는 배경 영감이자 의미론 모듈의 연구 동기로 위치한다 — 검증 도구로서의 유용성과 독립적으로 평가되어야 한다.

## [1] 사용 언어

**이번 단계: Python.**
- framework MVP를 가장 빠르게 *동작·실증*한다. 입시·연구 양면에서 가치는 "논리적 상대성이 실제로 작동하는 인공물"에 있지 언어 위신에 있지 않다.
- 유리학의 본질은 *논리 정의 파일(세계)을 반복 실험*하는 것 — Python의 빠른 반복 루프가 적합.
- 신뢰 경로의 의존성을 없애기 위해 파서까지 직접 구현(lark 등 미사용). 신뢰 경계가 깨끗해진다.

**커널 경화 단계 권장: OCaml 또는 Rust.**
- 대수적 자료형(sum type) + 망라적 패턴매칭 → 신뢰 코어가 *구성적으로* 옳다(빠뜨린 경우를 컴파일러가 잡음). 증명검증기·타입이론 연구의 사실상 표준(Coq=OCaml).
- de Bruijn 기준(수백 줄의 감사 가능한 커널)에 가장 잘 맞는다.
- 권장 경로: **지금 Python → 커널만 OCaml/Rust로 포팅**(파서·CLI·LLM은 Python 유지). 커널은 작아서 포팅 비용이 낮다.

**비채택**
- C++ / Java: 장황하고 합타입·패턴매칭이 없어 AST·규칙 표현이 취약. 신뢰성 검증이 어렵다.
- Go: 합타입·패턴매칭 부재(인터페이스+타입스위치로 AST를 다루면 지저분).
- JavaScript: 무타입 — 신뢰해야 하는 커널에 부적합.
- **Stele 자체로 self-host**: 매력적이나 시기상조. 특히 *Lean으로 self-host*하는 변형은 Lean이 단일 고전 기초에 헌신하므로 유리학의 "단일 기초 거부"와 정면 충돌한다.

## [2] 산출물 유형

**복합형.** 핵심 = 동작하는 수리논리 검사기(이번 산출물). 거기에 결정·근거·결과 문서를 붙인다.

- **exe / 패키징**: 배포 세부사항이라 보류. CLI(`python -m stele.cli`)로 충분.
- **LLM / SLM, 머신러닝·데이터사이언스**: 아키텍처상 *마지막* 단계(Phase 8). 지금 손대면 MVP 규율과 신뢰 모델(LLM은 untrusted, 커널이 재검사)을 깬다.
- **언어학·철학 논문**: matrix·세계격자 단계가 안착하면 나올 근거리 산출물. 주제 후보: "logical pluralism made executable — 논리 프레임워크로서의 Stele". *이 코드가 그 논문의 실증 인공물*이 된다.
- **수리논리 PL 개발**: 본 프로젝트가 정확히 이것이며, 다음 단계들의 척추다.

## 입시 관점 (정직하게)

- 독창적 수학철학(유리학) + 그것을 *실증하는 동작 코드*의 결합은 드물고 기억에 남는다. 추상 선언이 아니라 "같은 증명, 두 세계, 두 판정"을 명령행에서 보일 수 있다.
- 같은 산출물의 재포장 각도: 수학과 = 기초론·논리 다원주의 / CS = 형식검증·PL / 철학 = 논리철학.
- **과장 금지**: 현재 범위는 *명제논리 단편 + 다치 의미론 실증*이다. 1차 논리·세계 격자·정리증명은 아직 미구현(로드맵에 있음). 이 경계를 정확히 말하는 것이 신뢰를 만든다.
