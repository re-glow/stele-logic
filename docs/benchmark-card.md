# Benchmark Card — Stele Proof-Verification ML Baseline

## Overview

| Field | Value |
|-------|-------|
| Task | Binary proof-validity classification + multi-label diagnostic-code prediction |
| Model type | Multinomial Naive Bayes (stdlib, no external dependencies) |
| Authoritative verifier | `stele/kernel.py` (symbolic, trusted) — the ML model is UNTRUSTED |
| Corpus type | Synthetic, template-based |
| Committed sample size | 40 records (seed=0) |
| In-memory training set | 400 records (seed=0, generated on demand) |

---

## Dataset

### Generation

All records are generated by `bench/generate.py`. Generation is fully deterministic
given `(--corpus, --n, --seed, --shard-size)`. No human-annotated data is used.

Corpus families and proportions (`--corpus all`):

| Family | Proportion | Description |
|--------|-----------|-------------|
| `prop_nd` | 60% | Natural-deduction proofs in propositional logic |
| `definition_use` | 20% | Proofs with formula definitions (valid and with UndefinedDefinition errors) |
| `diagnostic_errors` | 20% | Proofs with injected errors (MissingHypothesis, InvalidTransition, etc.) |

### Committed sample

`bench/generated/sample/` contains 40 records (seed=0, 2 shards of 20).
This sample is used for smoke-test evaluation only. It is **not** representative
of a real-world proof corpus.

To regenerate the sample:
```bash
python bench/generate.py --corpus all --n 40 --out bench/generated/sample --seed 0 --shard-size 20
```

### Labels

All labels (`expected_valid`, `expected_codes`) are produced by running the
symbolic checker and diagnostic module against each generated proof. No label
is manually entered or fabricated.

---

## Diagnostic Codes

Six surface diagnostic codes appear in the v1 generator:

| Code | Description |
|------|-------------|
| `InvalidTransition` | Rule application does not match available hypotheses |
| `MissingHypothesis` | A referenced label is undefined |
| `UndefinedDefinition` | A definition name references an undefined symbol |
| `UndefinedSymbol` | A propositional variable used before introduction |
| `UnusedAssumption` | An assumption is never used in the proof |
| `UnsupportedConclusion` | The conclusion does not follow from the stated steps |

`CircularDependency` and `TypeMismatch` are defined in the diagnostic system
but not yet surfaced by the v1 corpus generator.

## Known Limitations

1. **Template-level leakage**: Records within a corpus family share proof
   templates. A train/test split by record ID does not guarantee
   template-level separation. Treat split metrics as indicative, not conclusive.

2. **Small corpus**: The committed sample is 40 records. In-memory training
   uses 400 records. Both are far too small to draw general conclusions about
   model capability.

3. **Synthetic distribution**: All proofs are generated from a small set of
   templates. This distribution does not reflect real user proofs.

4. **Naive Bayes limitation**: The bag-of-words model has no structural
   understanding. It learns surface token co-occurrences, not proof structure.

5. **Class imbalance**: Some diagnostic codes appear rarely in the corpus,
   leading to zero support in small test splits and undefined/zero F1.

## Measured Metrics (latest baseline_report.json)

See `stele_ml/reports/baseline_report.json` for the current measured values.

Do not read benchmark-card.md metrics as current — always check the report file
which is regenerated by running the pipeline.

To regenerate:
```bash
# Train on 400 in-memory examples
python -m stele_ml.train --n-generated 400 --out stele_ml/artifacts/baseline \
    --report stele_ml/reports/training_report.json

# Evaluate on committed 40-example sample
python -m stele_ml.eval \
    --model stele_ml/artifacts/baseline \
    --data bench/generated/sample \
    --report stele_ml/reports/baseline_report.json
```

## Honesty Constraints

- **Do not cite numbers from this document as "current accuracy."**
  Read `stele_ml/reports/baseline_report.json` instead.
- **Do not treat ML predictions as verified.** Only `stele/kernel.py` verifies.
- **Do not extrapolate from small-sample metrics** to claims about the model's
  general capability.
- Any metric in a report file must be computed by the evaluation pipeline,
  not entered manually.

## Reproducibility

Full pipeline reproduction from scratch:

```bash
# 1. Regenerate committed sample corpus
python bench/generate.py --corpus all --n 40 --out bench/generated/sample \
    --seed 0 --shard-size 20 --validate

# 2. Build train/dev/test split
python stele_ml/build_dataset.py \
    --source bench/generated/sample \
    --out stele_ml/data/sample_split \
    --seed 0

# 3. Train baseline on larger in-memory corpus
python -m stele_ml.train --n-generated 400 --out stele_ml/artifacts/baseline \
    --report stele_ml/reports/training_report.json

# 4. Evaluate on committed sample
python -m stele_ml.eval \
    --model stele_ml/artifacts/baseline \
    --data bench/generated/sample \
    --report stele_ml/reports/baseline_report.json
```

All steps are deterministic given the same seeds. No network access required.
No GPU required. No external ML dependencies required (pure Python stdlib).
