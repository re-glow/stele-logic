# Stele — Known Limitations

This document lists known limitations of Stele v1.4. Each limitation is stated
plainly with its scope and any relevant future-work note.

## Proof language (Stele-Light)

**Propositional surface only.** The proof-script language (`theorem`, `assume`,
`have`, `conclude`, etc.) covers propositional logic. First-order quantifiers
(`forall`, `exists`) are not available at the script level. The proof-term core
(`stele/core/fol.py`) has an experimental FOL fragment, but it is not exposed
in the surface language.

**Natural-deduction format required.** Every inference step must be written
out explicitly with a rule name and references. Stele does not fill in steps,
search for proofs, or infer omitted arguments.

**No equality or function symbols.** The proof language has no built-in
treatment of equality (`=`), function application, or term rewriting.

## Kernel and metatheory

**Propositional only.** The kernel checks propositional natural-deduction proofs.
There is no FOL or higher-order logic at the kernel level.

**Metatheory is not machine-checked.** Claims about subject reduction,
normalization, and consistency in `docs/metatheory.md` are supported by
regression tests and proof sketches. They have not been verified in Lean,
Coq, or Agda.

**Python import isolation.** The kernel/matrix separation is enforced by
test assertions on Python's import graph, not by language-level module
boundaries or a separate process. A malicious or buggy import of `matrix.py`
inside `kernel.py` would not be caught at the language level.

## Classical proof terms

**No λμ-calculus.** Classical rules (`dne`, `lem`, `pbc`) are checked by
the kernel at the proof-script level but do not have corresponding proof-term
witnesses. The proof-term core covers the intuitionistic fragment only.
`stele/core/classical_experimental.py` provides a Gödel–Gentzen negative
translation (formula-level only), not a full classical proof-term calculus.

## Kripke semantics

**Bounded search.** `stele/kripke.py` searches Kripke frames exhaustively
up to a configurable bound (default ≤4 worlds). Absence of a countermodel
within the bound does not imply the formula is intuitionistically valid —
there is no completeness theorem implemented. Only propositional IPL.

## Certificates and minicheck

**Code-level isolation only.** `stele/minicheck.py` does not import the kernel
or parser, but it runs in the same Python process and runtime as the main checker.
It provides code-level isolation, not process-level or hardware-level isolation.
A future Rust or OCaml port would provide stronger guarantees.

**Not formally verified.** Minicheck's correctness is supported by regression
tests only.

## Matrix semantics

**Three matrices.** K3 (strong Kleene), LP (Logic of Paradox), and Boolean
are the only built-in matrices. Extending to other many-valued logics requires
adding matrix data to `stele/matrix.py`.

**Separate from proof scoring.** Matrix soundness checks (`stele/matrix.py`)
are diagnostic and do not affect the kernel's validity verdict.

## Browser Studio

**Pyodide CDN required on first load.** The browser Studio downloads the
Pyodide runtime (~8 MB) from CDN on first visit. It is cached by the browser
after that. Full offline mode (self-contained, no CDN) is not implemented.

**Pyodide startup time.** First-visit load takes a few seconds. Subsequent
visits load from cache.

## ML baseline

**Synthetic corpus only.** `stele_ml/` is trained on a small synthetic corpus
generated by the repo's own corpus generator. It is not trained on real proof
corpora and has not been evaluated on external benchmarks. It is an optional,
isolated, experimental feature — not part of the trusted checking path.

## Lean bridge

**Lean 4 required.** `stele_lean/` requires a local Lean 4 installation.
It is a one-way advisory bridge that maps Lean 4 error patterns to Stele
diagnostic codes. No round-trip export, no Lean proof generation.
Propositional fragment only. Experimental.

## Proof-state hints

**Untrusted and structural only.** `stele/proofstate.py` provides rule-applicability
suggestions based on structural pattern matching of the current proof context.
All hints carry `trusted=False`. There is no proof search, no ML, and no guarantee
that a hint leads to a valid proof. All suggestions must be kernel-rechecked.

## Whitepaper

**Draft.** The technical whitepaper (`docs/whitepaper.md`, `paper/stele-whitepaper.tex`)
is a draft that has not been peer-reviewed.

## Future work (conservative roadmap)

1. FOL quantifiers at the proof-script surface level.
2. Structural rule policies (weakening, contraction, exchange) for linear and
   relevance logic fragments.
3. Process-level certificate isolation (Rust/OCaml minicheck).
4. Machine-checked metatheory (Lean/Coq/Agda export) — distant future.
5. Full offline browser mode (self-contained Pyodide).
