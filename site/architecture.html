Trust & Architecture

Architecture & Trust Model

Stele's correctness guarantee rests on a single small component: kernel.py. Every other layer — the parser, the Studio UI, diagnostics, proof-state hints, certificates, the ML baseline, the Lean bridge — is either explicitly untrusted or optional. They may be helpful, but they cannot change whether the kernel accepts or rejects a proof.

This is not a performance claim or a marketing claim. It is a structural invariant enforced by the codebase and checked by tests. kernel.py imports nothing from the standard library except built-in types; no third-party dependency; no semantic reasoning. Everything you need to audit Stele's correctness is in one file.

The architecture page explains what each layer does, which layers are trusted, and where the trust boundaries lie. It also covers the certificate/minicheck re-verification path, browser-local execution via Pyodide, and the six invariants the codebase enforces.

Trust layers

Trusted core Kernel
Pure syntactic rule matcher. Checks each have / conclude step against the RuleSchema data in logic.py. No semantic reasoning. No external imports. The sole source of proof validity verdicts. De Bruijn criterion: auditable in one reading.
stele/kernel.py
Trusted Parser
Hand-written recursive-descent tokenizer and parser. No external parser library (no lark, no PLY). Stdlib only. Produces the AST consumed by the kernel. Correctness is covered by 200+ parser tests.
stele/parser.py
Trusted Logic rule data
Rules are data — RuleSchema frozen dataclasses — not kernel logic. Five built-in logic objects: intuitionistic_prop, classical_prop, minimal_prop, etc. Adding a new logic means adding rule data, not editing the kernel.
stele/logic.py
Untrusted Structural diagnostics
Multi-pass analysis: undefined symbols, missing hypotheses, circular dependencies, unused assumptions, invalid transitions. Advisory only — never imported by the kernel. Nine stable diagnostic codes.
stele/diagnostics.py
Untrusted Proof-state hints
Rule-suggestion hints based on structural pattern matching of the current proof context. All hints carry trusted=False. No proof search. No ML/LLM. Hints must be verified by the kernel before use.
stele/proofstate.py
Experimental Certificates + minicheck
Kernel-gated certificate emission: the kernel verifies the proof first, then certificate.py serialises it to a versioned JSON format. minicheck.py provides an independent re-verification path — same process, separate code, no kernel import.
stele/certificate.py · stele/minicheck.py
Optional ML baseline
Multinomial Naive Bayes classifier trained on a synthetic proof corpus. Completely isolated: no stele import, no effect on kernel correctness. Disabled unless explicitly installed.
stele_ml/
Optional Lean 4 type-error bridge
Parses Lean 4 error output and maps type mismatch patterns to Stele diagnostic codes. One-way, advisory bridge — no round-trip export, no Lean proof generation, no kernel involvement.
stele_lean/
Enforced invariants

Six structural invariants

These invariants are maintained by the codebase and checked by automated tests. They are not aspirational — violation causes test failure.

  1. 1

    Trust boundary = kernel.py only

    The kernel performs purely syntactic, decidable matching. It makes no appeal to the semantics of any target logic. It stays small and auditable (de Bruijn criterion). Semantic checks are never pulled into the kernel.

    Test: test_regression_invariants.py
  2. 2

    Rules are data, not kernel logic

    Inference rules live in logic.py as RuleSchema dataclasses. The kernel consumes rule data; it does not contain rule logic. Adding a new logic means adding a Logic object — not touching the kernel.

    See: stele/logic.py · CLAUDE.md invariant 2
  3. 3

    Zero runtime dependencies

    The trusted core modules use only the Python standard library. No lark, no PLY, no numpy, no third-party packages in the proof-checking path. Tests require only pytest. Optional extensions (stele_ml) are isolated from the trust path.

    CLAUDE.md invariant 3
  4. 4

    K3/LP matrix definitions are locked

    The K3 and LP truth tables match specific Kleene/Priest conventions. Specifically: I→F = I and F→I = T in K3. A regression test (test_k3_imp_table_matches_manifesto) locks these values; any accidental change fails the test suite.

    Test: test_k3_imp_table_matches_manifesto
  5. 5

    Honesty in documentation and error messages

    Rule-relativity is relative to rule availability, not arbitrary interpretation. Underivability is a meta-claim that the proof checker does not establish directly. Unmeasured metrics and unverified claims must not appear in documentation.

    CLAUDE.md invariant 5
  6. 6

    Proof mode and matrix mode are separated

    kernel.py does not import matrix.py. matrix.py does not import kernel.py. The proof validity verdict and the semantic soundness check are computed by entirely independent code paths.

    Test: test_regression_invariants.py — kernel↔matrix separation
Re-verification

Certificates & minicheck Experimental

After the kernel verifies a proof, you can export a proof certificate and re-verify it with minicheck — an independent checker that shares no code with the main kernel.

What a proof certificate is

A proof certificate is a versioned JSON file that encodes a verified proof in a normalised format. It records the theorem name, the logic, the conclusion formula, and each proof step (assume, suppose_open, suppose_close, have, conclude) with explicit rule and references.

The certificate is emitted only after the kernel verifies the proof. A proof that the kernel rejects cannot produce a certificate.

What minicheck is

minicheck.py re-verifies a certificate JSON file using independent Python code — no parser, no kernel, no diagnostics imported. It reconstructs the proof state from the certificate's step list and applies the same syntactic rule checks the kernel would apply.

Limitations

Minicheck runs in the same Python process as the kernel — it is code-level isolation, not process-level or hardware-level isolation. Minicheck has not been formally verified. Its correctness is supported by regression tests only. A future Rust/OCaml port would provide stronger isolation.

CLI

# Emit certificate
python -m stele.cli cert examples/imp_self.stele \
  --logic intuitionistic_prop

# Save to file and re-verify
python -m stele.cli cert examples/dne.stele \
  --logic classical_prop --out dne.json
python -m stele.cli minicheck dne.json
Browser Execution

Browser-local via Pyodide Stable

Studio runs the full Stele proof checker in your browser via Pyodide/WebAssembly. No proof text is sent to any server. No backend. No installation required.

Pyodide is a port of CPython to WebAssembly. On first visit, the browser downloads the Pyodide runtime (~8 MB, cached). The Stele package is bundled as a zip at build time and unpacked in the browser's virtual filesystem.

The stele.browser module exposes thin JSON wrappers over the core functions. These wrappers are called from stele-pyodide.js, which bridges JavaScript and Python. The kernel runs inside Pyodide with the same code path as the local CLI.

The browser execution model means: the kernel you see in the repository is the kernel that runs in Studio. No server-side substitution.

Build process

tools/build_pyodide_site.py packages the Stele source into stele_source.zip for browser delivery. ML (stele_ml/) and Lean (stele_lean/) bridges are excluded from the browser bundle.
