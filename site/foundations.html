What Stele implements

Implemented components Stable

These components exist, pass tests, and are available in the current release. Claims about them are claims about running software.

Stable Multiple logic rule sets

Five built-in propositional logics: intuitionistic, classical, minimal, K3-logic, LP-logic. Rules are data — RuleSchema objects — not kernel logic. Adding a new logic does not require touching the trusted core.

stele/logic.py · stele/kernel.py
Demo / Diagnostic Matrix diagnostics

K3 (strong Kleene), LP (Logic of Paradox), and Boolean matrices compute rule soundness (⊨) as a diagnostic tool. Separate from the kernel (⊢). Shows how the same rule behaves differently in three-valued contexts.

stele/matrix.py · stele/world.py
Experimental Kripke countermodels

Bounded finite-frame search for intuitionistic propositional logic. Demonstrates that LEM and DNE fail intuitionistically by constructing explicit finite Kripke models. Bounded — no countermodel ≠ valid.

stele/kripke.py
Experimental Proof certificates + minicheck

After kernel verification, a versioned certificate JSON can be emitted. An independent re-verifier (minicheck.py) checks it without importing the kernel. Code-level isolation; process isolation is future work.

stele/certificate.py · stele/minicheck.py
Untrusted Diagnostics + hints

Multi-pass structural analysis and rule-suggestion hints. Advisory only — never imported by the kernel. All hints carry trusted=False. No proof search, no ML in the hint path.

stele/diagnostics.py · stele/proofstate.py
Stable Browser-local Studio

The full proof checker runs in your browser via Pyodide/WASM. No backend. No data transmitted. The same kernel that runs locally is the kernel that runs in the browser.

site/studio.html · stele/browser.py
Not implemented

Stele does not implement: a full Yurihak logic, a complete mathematical foundation, automatic theorem proving, full mathematical relativism, equality/function-symbol FOL semantics, proof-search, or a machine-checked philosophical system. These are listed as research motivation or future formalization below.

Research Motivation

Yurihak / 유리학 as research program Motivation

Yurihak is a philosophical and foundational research program — not a logic system currently implemented in Stele. It shapes the questions Stele is built to investigate.

Stele grew out of an interest in what happens when the rules of reasoning are treated as an explicit, variable parameter rather than a fixed background. Classical logic is one choice; intuitionistic logic is another. Multi-valued matrices like K3 and LP represent further choices. Each choice admits some inferences and blocks others.

This variability is not merely aesthetic. If different rule systems can each be coherent, the question of which system to use becomes a substantive one about what kind of reasoning a given domain requires. Yurihak — 유리학 (有理學), a term from Jaehwan Kim's foundational research — investigates this space: the conditions under which rule systems, semantic interpretations, and model-theoretic structures can vary in principled ways.

Stele operationalizes one piece of this inquiry: by making rule choice explicit at the proof level (the --logic flag and the RuleSchema data structure), and by separating syntactic derivability (what the kernel checks) from semantic validity (what the matrix layer checks), Stele creates a controlled environment for observing how the same formula behaves under different logical frameworks.

This is a motivation, not a proof. Stele does not prove mathematical relativism. It does not implement Yurihak as a formal system. It provides a platform on which formal Yurihak concepts could eventually be specified, tested, and checked — once they are rigorously defined.

What Stele shows, concretely
  • The same proof is accepted by classical_prop and rejected by intuitionistic_prop — demonstrating that rule choice determines what counts as valid.
  • Under K3 semantics, some classically-sound rules are no longer sound — demonstrating that semantic choice interacts with rule choice.
  • Kripke frames provide explicit countermodels to LEM and DNE — demonstrating that these are not universal truths, but rule-relative ones.
  • The proof-term layer separates the intuitionistic fragment (with Curry–Howard witnesses) from classical rules, making the boundary computationally visible.
What this is not

These demonstrations are software behaviors, not philosophical proofs. Stele does not prove that mathematical relativism is correct, or that Yurihak is a complete foundation of mathematics. The tool investigates these questions; it does not settle them.

What comes next

Future formalization roadmap Future

A conservative, step-by-step path toward formalizing Yurihak-inspired concepts in Stele. None of these steps are currently implemented.

This is a plan, not an implementation

The items below describe future work. They are aspirations, not current features. Any formal Yurihak logic will be developed outside the trusted kernel until it has a complete specification, formal semantics, and soundness evidence.

  1. 1

    Define a small Yurihak-inspired object language

    Start with syntax only: formulas, connectives, and structural rules, written as a spec document. No implementation until the syntax is stable.

  2. 2

    Specify rule sets separately from semantics

    Following Stele's existing architecture, define Yurihak rules as RuleSchema data before defining any semantic interpretation. This respects the architectural invariant: rules are data, not kernel logic.

  3. 3

    Give matrix / Kripke / world interpretations only after syntax

    Semantic models for Yurihak-inspired rule sets should be specified after the proof-level syntax is settled. Use existing matrix and Kripke infrastructure where it applies; extend or isolate where it does not.

  4. 4

    Add worked examples and test them

    Each proposed Yurihak rule must have a worked proof example accepted by the kernel, and a failing example that shows where the rule differs from existing logics. Examples must pass pytest.

  5. 5

    Test soundness claims before asserting them

    Use rule_soundness() against candidate matrices. Document which rules are sound under which interpretations. Regression-test these claims. Do not assert machine-verified soundness unless Lean/Coq proofs are produced.

  6. 6

    Keep outside the trusted core until mature

    Any Yurihak logic lives in an isolated module outside kernel.py and logic.py during development. Only stable, well-tested rule sets are candidates for inclusion in the built-in logic set.

Where this fits in the trust model

Philosophical motivation is untrusted

The trust boundary in Stele is maintained regardless of the philosophical context that motivates the project.

The kernel doesn't know about Yurihak

kernel.py performs pure syntactic rule matching. It does not know that Stele is motivated by logical pluralism. It does not reason about philosophical positions. It checks whether each proof step matches its cited rule — nothing more. Philosophical narratives belong to the motivation layer, not the trusted core.

Semantic tools provide countermodels, not metaphysical proof

The matrix semantics layer and Kripke countermodel search are Untrusted / Demo diagnostics. When the Kripke search finds a countermodel for LEM, it demonstrates that LEM is not intuitionistically valid — it does not prove a philosophical claim about the nature of truth. These are technical results about specific formal systems.

ML and hints are untrusted

The ML baseline classifier and proof-state hints have no influence on kernel correctness. If ML guidance ever points toward a Yurihak-inspired inference rule, that rule still needs to be formally specified as a RuleSchema and verified by the kernel before it means anything.
