Research & Whitepaper

Stele is a formal verification tool motivated by the research problem of supporting multiple logical frameworks in a single proof-checking infrastructure. This page connects the implementation to its theoretical context.

Metatheory status

Metatheory claims (subject reduction, normalization, consistency) are supported by regression tests and proof sketches — not by machine-checked proofs. Tests are evidence, not proof.

Research Motivation

Mathematical logicians have long known that no single logic captures all valid reasoning. Intuitionistic logic rejects the law of excluded middle; paraconsistent logics tolerate contradictions without explosion; relevance logics require genuine connection between premises and conclusion. The question of which logic to use is not settled — it is a live research problem in the philosophy and foundations of mathematics.

Stele's design responds to this situation concretely: rather than hard-coding classical propositional logic, it lets you declare which logic applies to a given proof via a --logic flag. The same proof script that passes under classical logic can be rejected under intuitionistic logic, exactly as the theory predicts. Rules are data (RuleSchema objects in logic.py), not hard-wired kernel branches.

The background inspiration is 유리학 (Yurihak) — a research programme exploring the boundaries of logical pluralism. Stele is not a formalization of Yurihak, and makes no claims on its behalf. It is a proof-checking infrastructure that the pluralist stance motivated: an engine whose logic can vary without changing the trusted core.

Scope note

Stele operates at the propositional level. Quantifier support (∀, ∃) exists in the proof-term layer (stele/core/fol.py) but is not yet exposed in the surface proof language. Logical pluralism at the first-order level is future work.

Relation to Proof Assistants

Stele is a proof checker, not a proof assistant or theorem prover. The distinction matters.

Lean 4 / Coq / Agda
Full proof assistants with dependent type theories, tactic languages, metaprogramming, and libraries spanning thousands of theorems. The user supplies high-level proof scripts; the system fills in details, searches for unifiers, and generates proof terms automatically.
Isabelle/HOL
Higher-order logic framework with Isar structured proofs and a large industrial-strength library (Archive of Formal Proofs). Supports automated reasoning via Sledgehammer. Full foundational commitment to HOL.
Stele
Proof checker for natural-deduction, propositional scope, and explicitly user-supplied rule sets. No proof search. No tactics. No metaprogramming. Every inference step must be written out and named. The kernel judges; the user constructs.

The de Bruijn criterion — the standard that a proof can be independently re-checked by a small, auditable kernel — is central to Stele's design. stele/kernel.py is the only trusted component. Everything else (diagnostics, hints, graph, Kripke) is explicitly labelled "untrusted" and does not affect the validity verdict.

What Stele deliberately is not: it does not search for proofs, does not fill in gaps, and does not export to Lean or Coq (that is listed as future work with appropriate caveats). It is a pedagogical and research tool for making the structure of natural-deduction arguments explicit and machine-checkable.

Transparency

What Stele claims — and doesn't

Evidence values are measured from the repository. Limitations are stated plainly — no inflation.

Claim Evidence Limitation
Subject-reduction holds for STLC fragment Property-based tests (Hypothesis); proof-term typing tests Not machine-verified in Lean/Coq; Hypothesis tests are regression, not formal proof.
Classical propositional logic is sound test_rule_soundness_classical_prop; all Boolean matrix checks pass Soundness is checked against the built-in matrix, not an independent semantic definition.
Intuitionistic logic is sound (bounded Kripke) Bounded countermodel search; Kripke tests; 15 curated examples Bounded search: countermodels up to width 3 only. Not a completeness proof.
Kernel is the only truth-producing path test_regression_invariants enforces import isolation (kernel ↔ matrix) Python's import system does not enforce isolation at the language level; it's a convention enforced by tests.
Proof-term elaboration is conservative elaborate.py crosscheck tests; term-check CLI examples Elaboration covers the intuitionistic fragment only; classical proofs are checked at rule level, not term level.
