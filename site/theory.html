Formal Foundations

Theory & Semantics

Stele is a proof checker, not a theorem prover. You write a natural-deduction proof step by step; Stele's trusted kernel checks each rule application against the formal inference rules you have selected. If a rule does not match, the proof fails — there is no search, no backtracking, no AI assistance.

The system has three independent layers. The proof checker kernel is the sole source of validity verdicts; it is small, dependency-free, and purely syntactic. An optional proof-term calculus layer lets you extract Curry–Howard witnesses from well-formed proofs, though this layer is experimental. A separate semantic diagnostics layer — matrices, world lattice, Kripke countermodel search — helps you understand why a formula behaves differently across logics; these tools are explicitly untrusted and cannot affect kernel verdicts.

This page explains those layers honestly. Features that are not yet stable are labeled Experimental. Metatheory claims backed only by regression tests — not machine-checked proofs — are described as "supported by regression tests", not "verified".

Proof Language

Stele-Light Stable

A structured natural-deduction language for propositional logic. Every proof is a sequence of explicit rule applications. The kernel checks each application; nothing is assumed or guessed.

Proof keywords

theorem NAME: Theorem header
assume L: φ Global assumption
suppose L: φ Subproof assumption
have L: φ by RULE refs… Derived step
conclude φ by L Final conclusion

Inference rules

15 rules in two sets. Rules are data (logic.py), not kernel logic.

copy mp imp_intro and_intro and_elim_left and_elim_right or_intro_left or_intro_right or_elim neg_intro neg_elim ex_falso dne ¹ lem ¹ pbc ¹

¹ classical only — rejected by intuitionistic_prop

Example proof — P → P

theorem imp_self:
  suppose h1: P
    have h2: P by copy h1
  have r: P -> P by imp_intro h1 h2
  conclude P -> P by r

Valid under both intuitionistic_prop and classical_prop. imp_intro discharges the suppose h1 assumption.

Classical-only example — DNE

theorem dne_consequent:
  assume h: not not P
  have r: P by dne h
  conclude P by r

Accepted by classical_prop; rejected by intuitionistic_prop — the dne rule is not in the intuitionistic rule set.

Proof mode

The proof checker kernel operates on the proof script alone. It makes no semantic claims: verifying P → P does not mean the kernel computed its truth value. The kernel checks that each have step follows from the cited references by the cited rule — nothing more.

Derivability vs. Validity

⊢ and ⊨ are distinct

Stele uses both symbols, but they are computed by entirely separate subsystems that do not share code.

Syntactic derivability (⊢)

Γ ⊢ φ means: the formula φ can be derived from assumptions Γ using only the rules in the selected logic. The kernel decides this by pattern-matching rule schemas against each proof step. No truth values are computed. The decision is purely structural.

This is what the kernel checks. A "Valid" verdict means every step matched its rule. It does not mean the formula is a tautology.

Semantic validity (⊨)

Γ ⊨ φ means: for every semantic valuation satisfying Γ under a given matrix, φ also holds. The matrix layer (matrix.py, world.py) computes this — as a diagnostic tool, not as a validity oracle.

A "Sound" matrix verdict means all rules preserve designated values. A "Unsound" result identifies which rules fail under which valuation. These are Demo diagnostics — the matrix layer is explicitly untrusted.

Curry–Howard

Proof-Term Calculus Experimental

An optional layer that extracts a typed λ-term (a Curry–Howard witness) from every well-formed intuitionistic proof script.

Under the Curry–Howard correspondence, propositions are types and proofs are programs. A proof of A → B corresponds to a function from A to B; a proof of A ∧ B corresponds to a pair; a proof of A ∨ B corresponds to a tagged sum.

The stele.core package implements this calculus for intuitionistic propositional logic. The elaborate module converts a verified Stele-Light script into a proof term, then stele.core.typing checks that term independently. The crosscheck_theorem function runs both checks in sequence.

β-reduction is implemented (stele.core.reduce), as are de Bruijn indices for capture-avoiding substitution. Metatheory claims (subject reduction, normalization, consistency) are supported by regression tests and proof sketches — they are not machine-verified.

Classical limitation

The proof-term calculus covers intuitionistic proofs only. Classical rules (dne, lem, pbc) are accepted at the proof-checker level but are excluded from the proof-term extractor (_CLASSICAL_RULES frozenset). The Gödel–Gentzen bridge is a separate experimental module.

Curry–Howard correspondence (excerpt)

Logic Type Term
A → B A → B Lam(x, A, body)
A ∧ B A × B Pair(a, b)
A ∨ B A + B Inl / Inr / Case
⊥ Empty Abort(t, type)
∀x. A(x) Π-type (approx) ForallIntro/Elim

CLI commands

# Cross-check a proof (kernel + type checker)
python -m stele.cli elaborate examples/imp_self.stele

# Check a single term
python -m stele.cli term-check "(fun x:P => x)" "P -> P"

# Normalize a term
python -m stele.cli term-normalize "(fun x:P => x) y"
First-Order Logic

FOL Proof-Term Fragment Experimental

Universal and existential quantifiers are supported in the proof-term layer. The surface proof language (Stele-Light) does not yet support FOL syntax — quantifiers are available only at the term level.

The proof-term calculus (stele.core.fol) extends the IPL fragment with ForallIntro, ForallElim, ExistsIntro, and ExistsElim constructors. These correspond to universal and existential quantifiers in the Curry–Howard reading.

Freshness conditions are enforced: ExistsElim rejects introductions where the witness variable appears free in the conclusion type. Formula-level α-equivalence uses a separate de Bruijn layer (to_debruijn_formula) to handle shadowing correctly.

Surface language limitation

The Stele-Light surface language does not yet support forall / exists syntax. Adding FOL quantifiers to the surface language and kernel is listed as a roadmap item. The current FOL support is proof-term level only (stele.core.fol), accessible via the Python API.

Classical Proof Terms

Gödel–Gentzen Bridge Experimental

An experimental module that translates classical propositional formulas into intuitionistic formulas via the Gödel–Gentzen double-negation translation, then type-checks manually-written proof terms for the translated formula.

The core claim being exploited is the metatheorem CPC ⊢ φ implies IPC ⊢ φN, where φN is the double-negation translation of φ (Gödel–Gentzen, 1933). This is a classical metatheory result; Stele does not machine-prove it.

The module (stele/core/classical_experimental.py) provides formula-level translation and an API to check proof terms for translated formulas. Representative cases (DNE, LEM, Peirce) are covered by regression tests.

Not implemented

λμ-calculus, callcc, continuation typing, classical normalization, automatic proof translation, and proof search are not implemented. The bridge is formula-level translation only — it does not automatically produce proof terms.

Multi-valued Semantics

Matrix Semantics Demo / Diagnostic

Three truth-value matrices — Boolean (classical), K3 (strong Kleene), and LP (Logic of Paradox) — let you inspect whether proof rules remain sound when truth can be indeterminate or paradoxical.

Boolean

Classical two-valued logic. All 15 rules sound. Baseline for comparison.

values: T, F
K3 (strong Kleene)

Three values: T, I (indeterminate), F. No designated I. LEM and DNE fail.

designated: {T} · I→F=I, F→I=T
LP (Logic of Paradox)

Three values: T, I, F. I is designated (paradoxical truths preserve). mp fails to preserve single-valuedness.

designated: {T, I}
Rule soundness vs. proof mode

Matrix soundness checks (⊨) are computed by matrix.py / world.py and are entirely separate from proof checking. A rule being "unsound under K3" does not make the kernel reject proofs — it only tells you whether the rule preserves designated values in that matrix. These are diagnostic tools for exploring semantic behaviour, not validity gates.

CLI commands 

# Check rule soundness under K3
python -m stele.cli soundness --logic classical_prop --matrix K3

# World lattice: P or not P across Γ, Γ+P, Γ+¬P
python -m stele.cli lattice "P or not P"
Intuitionistic Semantics

Kripke Countermodels Experimental

A bounded finite-frame search that looks for a Kripke countermodel for a given propositional formula under intuitionistic logic. Finding a countermodel proves a formula is not intuitionistically valid. Failing to find one within the bound is not a proof of validity.

A Kripke frame for intuitionistic propositional logic is a set of possible worlds with an accessibility relation. A formula is forced at a world if it holds at that world and at all reachable worlds. Intuitionistic validity means forcing at all worlds in all frames.

stele.kripke exhaustively searches frames up to a configurable bound (default: 3 worlds). If it finds a world where the formula fails to be forced, it returns a countermodel. The search is sound: any countermodel returned is a genuine Kripke countermodel for the intuitionistic semantics.

The search is not complete: a countermodel may require more worlds than the bound. A "no countermodel found" result within a small bound is not a proof of intuitionistic validity.

Known countermodels

Formula Result
P or not P Countermodel found (2 worlds)
not not P -> P Countermodel found (2 worlds)
((P->Q)->P)->P Countermodel found (3 worlds)
P -> P No countermodel (≤ 3 worlds) †

† Absence of countermodel within the bound is not a proof of validity.

CLI

python -m stele.cli kripke "P or not P"
python -m stele.cli kripke "not not P -> P" --max-worlds 4
Metatheory

Metatheory claims and status

Python regression tests are not machine-verified proofs. The table below uses precise language about what has and has not been shown.

Honesty statement

No metatheory claim in Stele is backed by a machine-checked proof (Lean/Coq/Agda). "Supported by regression tests" means specific cases pass. "Proof sketch" means a standard-metatheory argument is cited but not formalized in code. "Implementation invariant" means the code structure enforces the property, checkable by static inspection.
